# PRD: JD to KPI Converter

## 1. プロダクトビジョン

**「構造的経験格差を破壊し、誰もが定量的な努力でキャリアを切り拓ける世界へ」**
実務経験やチーム開発という定性的で曖昧な言葉によって生じている情報の非対称性を排除し、ポテンシャル層の挑戦を阻むバイアスをなくす。これは、一部への就業集中を防ぎ、最終的に「KAROSHI」を撲滅するための第一歩となるプロダクトである。

## 2. ターゲットユーザー

* **プライマリ:** 開発者自身（文系未経験からAIエンジニアを目指す学生）。
* **セカンダリ:** 独学でプログラミングを学んでいるが、企業の募集要項（JD: Job Description）を見て「自分には資格がない」と心理的障壁を感じているポテンシャル層。

## 3. 解決するペイン（課題）

1. **感情的バイアス:** 企業の採用ページにある「実務経験3年」「チーム開発経験」などの定性的な条件により、応募前に自分には能力がないと思い込む状態に陥る。
2. **証明手段の欠如:** 文系やソロ開発者には「ゼミや研究室でのチーム開発」という環境がなく、要件を満たすための具体的な代替アクション（何をどれだけやればいいか）が分からない。

## 4. コア体験（提供価値）

募集要項をコピペするだけで、AIが背後にある **「企業が本当に求めている技術的ハードル」** を逆算し、未経験のソロエンジニアが、個人開発で客観的に証明可能な『数値目標（KPI）』に変換して提示する。ユーザーは「お祈りされるかもしれない不安」から解放され、提示された数値をクリアするゲームに集中できる。

## 5. 機能要件 (MVP版)

最短でプロにデモを見せるための最小限の機能に絞り込む。

| 機能名 | 詳細 |
| --- | --- |
| **JD入力フォーム** | ユーザーが企業の募集要項（テキスト）を貼り付けて送信できるUI。 |
| **定性→定量 翻訳ロジック** | 入力されたJDをLLMに解析させ、「指標名」「目標数値」「GitHub等での証明方法」を導出する。 |
| **JSON構造化出力** | LLMの出力を単なるテキストではなく、必ず構造化されたJSONデータとして吐き出させる（バックエンドの安定性確保）。 |
| **KPIダッシュボードUI** | 生成されたJSONデータをパースし、直感的に分かりやすい表（またはリスト）として画面に出力する。 |

## 6. 非機能要件・技術スタック

* **フロントエンド / バックエンド:** `Streamlit` (Python)
* *選定理由:* 開発スピード最優先。1つのPythonファイルでUIからAPI連携まで完結でき、プロトタイプとして見栄えの良いものが即座に完成するため。


* **AIモデル:** `OpenAI API (gpt-4o)`
* *選定理由:* 複雑な要件の文脈理解と、高精度なJSON mode（Structured Outputs）による確実なデータパースを行うため。


* **インフラ・デプロイ:** `Streamlit Community Cloud` または `Render` (予定)
* *選定理由:* GitHubリポジトリと連携し、無料で即座にWeb上に公開・URL共有ができるため。



## 7. 開発マイルストーン (目安)

* **Day 1 (ロジック検証):** Pythonスクリプト上でOpenAI APIを叩き、サンプルの募集要項を「定性→定量」に変換するプロンプトの設計とJSON出力のテストを行う。
* **Day 2 (UI実装):** Streamlitを用いて、テキスト入力〜APIリクエスト〜結果のテーブル表示までの画面を作成する。
* **Day 3 (デプロイ＆検証):** Web上に公開し、実際の行きたい企業のJDを入力して自らの検証を行う。

## 8. 成功指標（Success Metrics）

本MVPにおける成功は、ユーザー数や売上ではない。

1. **自己効力感の獲得:** 開発者自身がこれを使った結果、「このKPIの数値なら、今の自分でも実装して証明できる」と私自身が心から思えること。
2. **プロの評価:** 社会に出て活躍されているEM等にデモURLと背景思想を送り、フィードバックまたは面談の機会を獲得すること。
